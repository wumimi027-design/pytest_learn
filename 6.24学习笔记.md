# Pytest 学习笔记（第1-2天）

## 一、pytest 是什么

pytest 是一个 **Python 测试框架**。它的作用是帮你自动执行所有测试用例，并统计、打印结果（多少通过、多少失败、失败在哪一行），不需要自己写重复的判断和打印逻辑。

---

## 二、测试函数的基本规则

- 函数名必须以 `test_` 开头，pytest 才会识别它是一个测试。
- 测试的核心写法是 **断言（assert）**：

```python
assert result["status"] == "success"
```

`assert` 的作用：断定某个结果应该是什么样的。如果断言成立，测试通过；如果不成立，**会直接抛出异常，中断当前测试函数**（不是"返回"一个错误，是直接报错并停在那一行）。

---

## 三、字典（dict）基础

字典是"键值对"结构，取值方式是 `字典名["键名"]`：

```python
person = {"name": "小明", "age": 20}
person["name"]   # 结果是 "小明"
```

（类似 JS 里的对象 `{key: value}`）

---

## 四、测试要覆盖"正常情况"和"异常情况"

测试的核心职责：**确认程序在各种输入下，结果都符合预期**——既要测试正确的输入（正向测试），也要测试错误的输入（反向测试），提前发现 bug，避免有问题的代码被发布。

```python
def test_login_success():
    result = login("admin", "123456")
    assert result["status"] == "success"

def test_login_fail():
    result = login("admin", "wrong_password")
    assert result["status"] == "fail"
```

---

## 五、parametrize：一个函数跑多组数据

**解决的问题**：如果用 `if` 判断多种情况，一旦某个判断失败，程序会立刻停止，后面的判断不会被执行到，看不到完整结果。

`parametrize` 让 pytest **把同一个函数完整跑 N 遍**，每一遍互不影响，哪个失败了单独标记，不会拖累其他几遍。同时大幅减少重复代码。

```python
import pytest

@pytest.mark.parametrize("password", [
    "wrong_password",
    "123",
    "",
    "ADMIN123",
])
def test_login_fail(password):
    result = login("admin", password)
    assert result["status"] == "fail"
```

运行结果中会显示成 4 个独立用例：
```
test_login_fail[wrong_password]
test_login_fail[123]
test_login_fail[]
test_login_fail[ADMIN123]
```

## 六、装饰器（Decorator）

装饰器写在函数上方，作用是：**给函数加一层额外的行为规则，不需要修改函数内部代码**。

- `@pytest.fixture` → 告诉 pytest "这个函数是数据提供者，不是普通测试函数"
- `@pytest.mark.parametrize` → 告诉 pytest "把这个函数重复执行 N 次，每次换一组参数"

**注意**：装饰器的名字是固定的（pytest 库定义好的，不能乱改）；但 fixture 函数自己的名字是自己起的，可以随意命名，只要和使用处对应上即可。

---

## 七、fixture：公共测试数据

**解决的问题**：避免多个测试函数重复写同样的准备数据，一处定义，多处复用。

```python
@pytest.fixture
def valid_user():
    return {"username": "admin", "password": "123456"}
```

使用方式：测试函数把 fixture 名字写进参数列表，pytest 自动识别并注入：

```python
def test_login_success(valid_user):
    result = login(valid_user["username"], valid_user["password"])
    assert result["status"] == "success"
```

**和 parametrize 参数的区别**：
- parametrize 的参数值，写在装饰器列表里
- fixture 的参数值，来自 pytest 自动查找同名 fixture 函数并执行后的返回值

---

## 八、conftest.py：fixture 的公共仓库

- **`conftest.py` 是固定文件名**，不能改名，pytest 会自动扫描这个文件。
- fixture 写在 `conftest.py` 里 → **同级目录下所有测试文件都能直接用**，不需要 import，不需要重复定义。
- fixture 写在某个测试文件自己内部 → 只有这个文件能用（类似局部变量 vs 全局变量）。

一句话总结两个文件的关系：**`test_login.py` 是考卷，`conftest.py` 是考前准备的工具箱。**

---

## 九、fixture 的 scope（作用范围/复用程度）

**解决的问题**：如果 fixture 准备数据很耗时（比如登录拿 token），每个测试都重新生成一次很浪费时间。`scope` 控制这份数据要不要在多个测试间共享。

scope 取值（共用范围从小到大）：
- `"function"`（默认）：每个测试函数都重新执行一次 fixture
- `"class"`：同一个测试类共用一份
- `"module"`：同一个文件共用一份
- `"session"`：整次运行，所有文件共用一份

**判断标准（核心逻辑）**：
> 测试只读数据、不会互相干扰 → 可以共用，用更大范围的 scope，省时间
> 测试会修改数据、可能互相干扰 → 必须独立，用 `scope="function"`，保证每次都是干净数据

**实测验证**：用 `print` 放进 fixture 里，配合 `pytest -v -s` 运行，能亲眼看到 `scope="function"` 时 print 出现 N 次（N = 用到该 fixture 的测试数），`scope="session"` 时只出现 1 次。

## 十、yield：fixture 的"准备 + 清理"机制

**解决的问题**：有些 fixture 不仅要在测试前准备数据，还要在测试后做清理工作（比如删除测试时插入的假数据、关闭连接）。

```python
@pytest.fixture
def valid_user():
    print("准备数据")
    data = {"username": "admin", "password": "123456"}
    yield data              # 把数据交给测试函数，暂停在这里
    print("测试结束，清理数据")   # 测试用完后，从这里继续往下执行
```

**关键点**：
- `yield data` 类似 `return data`，把值交给测试函数使用，但函数不会立刻结束，而是"暂停"在这一行。
- 等测试函数跑完，再回来从 `yield` 下面继续执行，做清理动作。
- `yield` 之后**不需要再写 return**，它自己就是完整机制，不是"yield + return"组合。
- 清理动作什么时候触发，由 `scope` 决定：
  - `scope="function"`：每个测试各自清理一次（清理次数 = 测试数）
  - `scope="session"`：等所有用到它的测试全部跑完，才统一清理一次

## 知识点之间的关系图（文字版）

```
pytest（框架）
 ├─ 测试函数：test_ 开头 + assert 断言
 ├─ parametrize：1个函数 → 跑N组数据（数据写在装饰器里）
 └─ fixture：公共准备数据（数据来自函数返回值）
      ├─ 写在哪：本文件内（局部）/ conftest.py（全局共享）
      ├─ scope：决定多少测试共用一份（function / class / module / session）
      └─ yield：让 fixture 同时具备"准备"和"清理"两个阶段
```
